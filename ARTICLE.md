# 用 AI + Drain3 打造智能运维日志分析平台：从「日志海」到「一句话结论」

> 项目地址：https://github.com/tyloryang/AI-logging-analyse
> 技术栈：Python / FastAPI / Vue 3 / Loki / Claude / Qwen3 / Drain3

---

## 一、场景：每天面对几十万条日志，你是怎么排查问题的？

相信每位做过后端或运维的同学都遭遇过这种场景：

**凌晨 3 点，监控告警响了。**

你打开 Grafana、跳转到 Loki，面对滚动不停的日志流，开始执行以下仪式：

```
{app="cloud-monitor"} |= "error" | limit 500
```

500 条日志刷出来。你盯着屏幕，逐行扫描，尝试在脑海中拼凑一个"出了什么事"的故事。

这个过程，**快则十分钟，慢则一个小时**。而且极度依赖个人经验——如果不是你负责的服务，你甚至不知道从哪里开始看。

更麻烦的是：

- **日志量爆炸**：微服务架构下，10 个服务、每服务每天几万条，总量轻松破百万
- **重复噪音多**：`Connection timeout to 10.0.1.5 after 30002ms` 和 `Connection timeout to 10.0.2.3 after 15443ms` 本质是同一个问题，却占了几百行
- **上下文割裂**：日志在 Loki，指标在 Prometheus，告警在钉钉，三个系统来回切换
- **日报全靠手写**：每日运维报告要汇总各服务状态、总结问题，通常是下班前最痛苦的半小时

**这些问题，本质上都是"信息密度低、人工处理成本高"的问题。**

---

## 二、问题拆解

把上面的痛点具体化，可以归纳为三个核心问题：

### 问题 1：原始日志太"啰嗦"，真正有价值的模式被淹没

一个生产系统每天产生 100 万条日志，但真正独立的**错误模式**往往只有几十种。
`timeout to 10.0.1.5`、`timeout to 10.0.2.3`、`timeout to 10.0.3.1`……这三条都是同一个问题，却占了三行。

你需要的不是原始日志，而是**「这类 timeout 错误今天出现了 523 次，涉及 cloud-monitor 服务」**。

### 问题 2：即使看懂了日志，"下一步该怎么做"仍然靠经验

不同人对同样的错误日志会有不同的判断：
- 新人：这个错误是什么意思？
- 有经验的工程师：这是下游服务响应超时，上次也出现过，是连接池配置问题
- SRE：需要立刻扩容还是等业务低峰期排查？

这种**经验的不可复制性**，导致问题响应速度极度依赖特定的人。

### 问题 3：信息分散，缺少统一的运维视图

日志在 Loki、指标在 Prometheus、告警在消息群里、日报在飞书文档里。
没有一个地方能让你**5 分钟内看清系统全貌**。

---

## 三、解决方案：Loki + Drain3 + AI 的三层架构

针对上面三个问题，我设计了一套三层处理流程：

```
原始日志（Loki）
     │
     ▼ 第一层：Drain3 模板聚类（压缩信息密度）
  日志模板列表
  "Connection timeout to <*> after <*>"  × 523
  "Failed to forward request: <*>"        × 312
     │
     ▼ 第二层：AI 分析（还原经验判断）
  结构化分析报告
  - 错误模式识别
  - 根因推断
  - 优化建议
     │
     ▼ 第三层：统一视图 + 自动推送（消灭信息孤岛）
  运维日报 → 飞书/钉钉
```

### 第一层：Drain3 算法 —— 把 10 万条日志压缩成 50 个模板

**Drain**（*An Online Log Parsing Approach with Fixed Depth Tree*）是一种基于固定深度前缀树的在线日志聚类算法。

它的核心思想是：**用结构（固定词 + 通配符 `<*>`）来表示一类日志，而不是单独记录每条。**

算法三步走：

```
步骤1：按 token 数量路由
  "Connection timeout to 10.0.1.5 after 30002ms"  → 长度=7 的子树

步骤2：按前缀路由
  第1个词 "Connection" → 进入 Connection 子树
  第2个词 "timeout"   → 进入 timeout 子树

步骤3：计算相似度
  候选模板: "Connection timeout to <*> after <*>"
  当前日志: "Connection timeout to 10.0.2.3 after 15443ms"

  逐 token 比对：
  Connection=Connection ✓  timeout=timeout ✓  to=to ✓
  <*> ≠ 10.0.2.3（已是通配符，跳过）
  after=after ✓  <*> ≠ 15443ms（已是通配符，跳过）

  相似度 = 5/7 = 0.71 ≥ 阈值(0.4) → 归入此模板 ✓
```

为了让 Drain 专注分析**消息体**，送入前会先做预处理：

```python
# 剥离时间戳: "2024-01-01T00:00:00Z ERROR main.go:42 Connection..."
#           ↓ 去掉时间戳
#             "ERROR main.go:42 Connection..."
#           ↓ 去掉级别
#             "main.go:42 Connection..."
#           ↓ 去掉调用位置
#             "Connection timeout to 10.0.1.5 after 30002ms"
```

**效果**：10 万条日志 → 几十个模板，人眼一扫就能看清问题分布。

### 第二层：AI 分析 —— 把模板变成可执行的建议

拿到结构化的日志数据后，将其送入 AI（Claude 或本地 Qwen3），AI 扮演"资深 SRE"的角色输出分析：

```
你是一名资深 SRE，请分析以下日志...

请提供：
1. 错误模式识别 - 识别主要错误类型和出现规律
2. 根因分析     - 推断可能的根本原因
3. 影响评估     - 判断对系统稳定性的影响程度
4. 优化建议     - 具体的修复和预防措施
5. 优先级排序   - 哪些问题需要立即处理
```

AI 输出通过 **Server-Sent Events (SSE)** 流式传输到前端，用户可以看到分析内容实时"打印"出来。

同时支持两种 AI Provider，通过环境变量切换，**数据不出内网**也能用：

```env
# 云端方案（推理能力强）
AI_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-xxx

# 本地方案（数据安全）
AI_PROVIDER=openai
AI_BASE_URL=http://192.168.x.x:8000/v1
AI_MODEL=Qwen3-32B
```

### 第三层：统一视图 + 自动推送 —— 消灭信息孤岛

每日运维日报包含：健康评分、错误 Top10、AI 分析摘要，并通过 APScheduler 按 cron 表达式**自动生成并推送**到飞书/钉钉，运维团队早上一进群就能看到昨晚的系统状况。

## 六、快速部署

### 最小化部署（5 分钟跑起来）

**前提**：有一个可访问的 Loki 服务，以及 Claude API Key 或本地模型服务。

```bash
# 1. 克隆项目
git clone https://github.com/tyloryang/AI-logging-analyse.git
cd AI-logging-analyse

# 2. 配置后端
cd backend
cp .env.example .env
```

编辑 `.env`：

```env
# Loki 地址
LOKI_URL=http://your-loki-host:3100

# AI 选择一种：
# ── 方案 A：Claude（云端，效果最好）
AI_PROVIDER=anthropic

# AI_PROVIDER=openai
# AI_BASE_URL=http://192.168.x.x:8000/v1

```

```bash
# 3. 启动后端
pip install -r requirements.txt
python main.py
# → API 服务: http://localhost:8000

# 4. 启动前端
cd ../frontend
npm install && npm run dev
```

## 七、技术选型说明

| 技术 | 选择理由 |
|------|----------|
| **Loki** | 原生 Label 体系天然支持按服务过滤；比 ELK 轻量，无需 ES 索引 |
| **Drain3** | 无监督、在线算法，无需预定义正则，适应各种日志格式 |
| **FastAPI** | 原生支持 async/await，SSE 流式输出零额外成本 |
| **APScheduler** | 内嵌进程调度，不依赖外部 cron；asyncio 调度器与 FastAPI event loop 共享 |
| **SSE（Server-Sent Events）** | 单向流式传输，比 WebSocket 实现更简单，满足 AI 文字流出需求 |
| **Provider 抽象层** | 一套接口同时支持 Claude 和 OpenAI 兼容接口，切换 AI 只改环境变量 |

## 八、总结

这个项目尝试回答一个问题：**能不能让运维工程师在 5 分钟内，从原始日志中得到可执行的结论？**

三层设计的分工很清晰：

- **Drain3** 负责降噪——把百万条日志压缩成有意义的模板
- **AI** 负责理解——把模板变成人能读懂的诊断报告
- **统一视图 + 推送** 负责传达——让结论在对的时间出现在对的地方

目前还有很多可以迭代的方向，比如：接入 Prometheus 获取真实节点状态、对模板的时序变化做异常检测、支持对话式日志查询等。
