# AIOps 智能运维平台设计文档

> 版本：v1.0 · 日期：2026-04-20 · 状态：已确认，待实施

---

## 一、理解摘要

### 构建目标
在现有 SxDevOps 平台基础上，构建以 **AI 为核心驱动力** 的 AIOps 智能运维平台，覆盖故障根因分析、告警降噪、多云资产统一管理、AI 辅助决策四大方向。

### 核心痛点（按优先级）

| 优先级 | 痛点 | 目标 |
|--------|------|------|
| P1 | 故障响应慢 | AI 自动关联日志/指标/Trace，给出根因 + 处置建议 |
| P2 | 告警风暴 | 智能降噪、告警聚合、抑制重复推送 |
| P3 | 多云资产散 | 统一资产视图、成本透明化 |
| P4 | AI 只是问答 | AI Agent 主动分析、生成工单建议、推送飞书 |

### 用户群体
- **主操作者**：SRE / 运维工程师（日常使用）
- **次要受众**：开发工程师（自助查询）、管理层（报告 + 大盘）

### 关键约束
- AI 执行边界：**纯建议模式**，所有操作由人确认，AI 不直接执行变更
- 数据规模：中规模（主机 200–1000，日志 10–100GB/天，团队 20–100 人）
- 技术栈延续：Python FastAPI + Vue 3，复用现有 Loki/Prometheus/SkyWalking

### 明确的非目标
- 不做 AI 自动执行变更（当前阶段）
- 不做自建监控采集（继续依赖 Prometheus/Loki）
- 不做 IM 工具替代（飞书继续作为通知渠道）

### 假设
- Loki / Prometheus 数据质量已可用，不需要大规模清洗
- 告警规则已有一定基础（Prometheus AlertManager 或类似）
- 多云真实接入 SDK 从 Phase 3 开始，Phase 1/2 先做框架

│  │ RCA Engine  │  │ Alert Dedup  │  │  Anomaly   │  │
│  └─────────────┘  └──────────────┘  └────────────┘  │

       ├──▶ 飞书机器人推送                  ◀── 飞书 Webhook 事件回调
       │     · AI 根因摘要卡片                   （用户在飞书 @ 机器人触发分析）
       │     · 告警聚合通知
       │     · 巡检报告推送

                       │

### 设计原则
- 中枢引擎是**纯后端模块**，直接在 FastAPI 内作为独立 service 层，不引入新中间件
- AI 分析是**按需触发**（告警到来、用户点击、定时巡检），不做常驻流式消费
- 飞书既是**通知出口**（推送告警/根因），也是**触发入口**（@ 机器人触发分析）

## 三、核心模块详细设计

### 模块一：根因分析器（RCA Engine）

**触发方式**：告警到来 / 用户手动触发 / 飞书 @ 机器人

**分析流程：**
```
告警事件进入
    │
    ▼
① 上下文采集（并行拉取，超时 10s）
    ├─ Loki：故障时间窗口内相关服务 ERROR 日志（最近 500 条）
    ├─ Prometheus：CPU / 内存 / 错误率 / P99 时序数据
    ├─ SkyWalking：异常 Trace + 慢接口 TopN
    └─ CMDB：受影响主机信息 + 最近变更记录
    │
    ▼
② 上下文压缩（避免超 Token 限制）
    · 日志去重 + 保留最高频错误模式
    · 指标只取异常时间段的峰值 + 趋势描述
    │
    ▼
③ AI 分析（流式 SSE 输出）
    · Prompt 模板：角色 + 数据 + 输出格式约束
    · 输出结构：根因摘要 / 影响范围 / 置信度 / 处置建议（3 条）
    │
    ▼
④ 结果落库 + 推送
    · 存入 data/rca_results.json
    · 推送飞书卡片 + 写入前端根因面板
```

**输出数据结构：**
```json
{
  "root_cause": "Redis 连接池耗尽导致订单服务超时",
  "confidence": 0.85,
  "affected_services": ["order-service", "payment-service"],
  "evidence": [
    "错误日志中 ConnectionPool timeout 出现 342 次",
    "Redis QPS 在 14:32 突增 3x"
  ],
  "suggestions": [
    "扩大 Redis 连接池上限（当前 50，建议 200）",
    "检查是否有连接未释放的慢查询",
    "考虑开启 Redis Cluster 分片"
  ]
}
```

### 模块二：告警聚合器（Alert Dedup）

**核心目标**：把同一根因触发的几十条告警收敛成 1 张卡片推送。

**聚合流程：**
```
原始告警流入（AlertManager Webhook → 本平台）
    │
    ▼
① 指纹计算
    · 相同 {alertname + service + 时间窗口(5min)} → 同一指纹
    · 指纹相同的告警合并，只保留第一条触发时间 + 最新一条
    │
    ▼
② 分组策略
    ├─ 按服务分组：同一服务的所有告警归一组
    ├─ 按主机分组：同一主机的资源类告警归一组
    └─ 按时间窗口：5 分钟内的同类告警强制合并
    │
    ▼
③ 抑制规则
    · 父告警触发时，子告警静默
    · 同一 RCA 推送后，15 分钟内不重复推送同类告警
    │
    ▼
④ 输出
    ├─ 飞书：聚合卡片（"共 17 条告警，根因已分析"）
    └─ 前端告警中心：告警组视图 + 展开原始告警
```

**告警状态机：**
```
NEW → GROUPED → ANALYZING → RESOLVED
              ↓
           SUPPRESSED（被父告警抑制）
```

**存储**：`data/alert_groups.json`

### 模块三：异常检测器（Anomaly Detector）

**触发**：定时任务每 5 分钟执行一次

**检测策略：**
```
① 拉取基线：Prometheus 过去 7 天同时段历史数据
② 三种判断策略
    ├─ 静态阈值：CPU > 90%，错误率 > 5%（配置在 settings）
    ├─ 动比判断：当前值 > 历史均值 × 3σ（适合流量/QPS 类）
    └─ 趋势判断：连续 3 点单调上升且斜率超阈值（适合内存泄漏类）
③ 严重度打分
    · P0 → 触发 RCA Engine + 推送飞书
    · P1 → 写入告警聚合器，等待人工关注
    · P2 → 只写前端告警中心，不推飞书
```

**约束**：不引入 Prophet / sklearn，纯 Python 实现，保持部署简单。基线数据不持久化，按需从 Prometheus 拉取。

---

### 模块四：多云资产管理器（Cloud Asset Manager）

**统一适配器接口：**
```python
class CloudAdapter:
    async def list_instances(self) -> list[CloudResource]: ...
    async def get_cost_summary(self) -> dict: ...
    async def test_connection(self) -> bool: ...
```

**统一资源数据结构：**
```python
CloudResource {
  id, name, type, region,
  status, spec, tags,
  cost_monthly, provider
}
```

**数据流：**
- 定时同步（每小时 / 手动刷新）
- 各云 SDK 适配器拉取 → 统一结构 → 存入 `data/cloud_assets.json`
- 前端按 provider / region / type 多维筛选

**接入阶段：**

| 阶段 | 内容 |
|------|------|
| Phase 3 | 阿里云（ECS / RDS / OSS），完成适配器框架 |
| 后续 | AWS / 腾讯云 / 华为云逐步接入 |
| 未来 | AI 分析闲置资源，给出成本优化建议 |

### 新增页面

| 页面 | 核心内容 |
|------|----------|
| 故障大盘 | 实时 P0/P1 告警列表、服务健康评分、一键触发 RCA |

- 根因分析卡片风格与飞书推送卡片保持一致

## 五、实施路线图

### Phase 1 · 告警治理（第 1–2 周）

**后端**
- AlertManager Webhook 接入端点
- 告警聚合器（指纹 + 分组 + 抑制规则）
- 飞书聚合卡片推送
- 告警状态机 + 持久化

**前端**
- 告警中心页面
- 故障大盘（基础版）

**验收标准**：同一故障触发的 20 条告警 → 飞书只收到 1 张聚合卡片

---

### Phase 2 · 根因分析（第 3–5 周）

**后端**
- RCA Engine（上下文采集 + 压缩 + AI 分析）
- 异常检测器（3σ + 趋势判断）
- RCA 结果落库 + 历史查询接口
- P0 告警自动触发 RCA 联动

**前端**
- 根因分析页面
- 异常检测页面
- 故障大盘升级（集成 RCA 入口）

**验收标准**：告警到来后 30 秒内，飞书收到 AI 根因分析卡片

---

### Phase 3 · 多云资产（第 6–8 周）

**后端**
- CloudAdapter 框架
- 阿里云 SDK 接入（ECS / RDS / OSS）
- 定时同步任务 + 资产入库
- 成本汇总接口

**前端**
- 多云资产页面升级（真实数据替换 Mock）

**验收标准**：阿里云 ECS 资源列表可见，月度费用可查

## 六、决策日志

| # | 决策 | 备选方案 | 选择原因 |
|---|------|----------|----------|
| 1 | AI 中枢 + 能力插件架构 | 渐进增强、事件驱动 | 跨域根因关联必须有中枢；中规模无需引入消息队列 |
| 2 | AI 纯建议模式，人工确认 | 半自动、全自动 | 当前阶段降低风险，建立信任后再放开自动化权限 |
| 3 | 按需触发分析，非常驻流 | 实时流式消费 | 中规模够用，避免常驻进程资源浪费 |
| 4 | 异常检测不引入 ML 库 | Prophet / sklearn | 保持部署简单，3σ + 趋势判断对中规模已足够 |
| 5 | 多云资产本阶段先接阿里云 | 四云同时接入 | YAGNI，先跑通框架再扩展 |
| 6 | 故障大盘作为默认首页 | 保留现有仪表盘 | SRE 最高频需求是看故障，而非总览图表 |
| 7 | 资产存 JSON 文件 | 引入 SQLite/PostgreSQL | 中规模够用，不增加运维负担 |
