# 飞书巡检告警公开 PDF 链接设计

## 背景

飞书巡检卡片会对告警主机数量、异常项和 AI 分析做长度限制。主机数量较多或巡检内容较长时，用户只能看到摘要，无法从消息中继续查看完整内容。

系统已经能够保存巡检报告 JSON，并通过 ReportLab 动态导出 PDF。本功能为所有飞书巡检通知增加公开 PDF 入口，让用户无需登录即可在浏览器中查看或下载完整巡检报告。

## 目标

- 覆盖全局定时巡检、分组定时巡检、分组手动巡检和报告页面手动推送。
- 飞书卡片统一显示“查看完整巡检 PDF”按钮。
- 全局通知链接完整巡检报告；分组通知链接对应分组的巡检报告。
- PDF 链接无需登录且不设置独立过期时间。
- 报告仍服从现有 `REPORT_RETENTION_DAYS` 保留策略；报告被清理后链接返回 404。
- 只公开巡检报告，不改变运维日报和慢日志报告的访问权限。

## 非目标

- 不把 PDF 上传到飞书、对象存储或 CDN。
- 不改变飞书卡片现有摘要和截断规则。
- 不改变钉钉通知。
- 不取消或延长现有报告清理策略。
- 不为历史巡检报告批量补发飞书消息。

## 方案选择

采用独立的巡检公开 PDF 路由，不直接公开现有通用导出接口。

未采用的方案：

1. 直接公开 `/api/report/{report_id}/export.pdf`：改动较少，但会连带公开日报和慢日志报告。
2. 预生成 PDF 并上传外部存储：会引入上传、存储和清理机制，超出本功能范围。

## API 设计

新增接口：

```text
GET /api/public/report/inspect/{report_id}.pdf
```

行为：

- 该路由不要求登录。
- `report_id` 必须符合现有报告 ID 的安全字符约束，且解析后的文件必须位于 `REPORTS_DIR` 内。
- 加载 `${REPORTS_DIR}/{report_id}.json`。
- 报告不存在、JSON 无效或 `type` 不是 `inspect` 时返回 404。
- 使用现有 `build_report_pdf()` 动态生成 PDF。
- 成功时返回 `application/pdf`，使用 `Content-Disposition: inline`，使浏览器优先打开预览并允许用户下载。
- PDF 构建失败时记录带报告 ID 的完整异常并返回 500。

现有需要登录的 `/api/report/{report_id}/export.pdf` 保持不变。

## 公开 URL 构造

增加一个统一的巡检 PDF URL 构造函数：

```text
{APP_URL}/api/public/report/inspect/{report_id}.pdf
```

约束：

- `APP_URL` 去除末尾 `/` 后再拼接路径。
- 飞书按钮必须使用绝对 HTTP(S) URL。
- `APP_URL` 未配置时返回空链接；巡检告警仍正常发送，但不添加按钮，并记录明确的配置警告。
- 不回退到 `localhost`，避免向飞书用户发送不可访问地址。

## 飞书卡片设计

### 完整巡检报告卡片

现有 `_build_feishu_inspect_card()` 已支持 `report_url`。当链接非空时，将按钮文案统一为“查看完整巡检 PDF”，按钮使用飞书交互卡片的 URL 跳转能力。

### 分组巡检告警卡片

为 `_build_feishu_group_inspect_card()` 和 `send_feishu_group_inspect()` 增加可选的 `report_url` 参数。链接非空时，在卡片末尾增加主按钮“查看完整巡检 PDF”；链接为空时不生成空按钮。

## 数据流

### 全局定时巡检

1. 采集全局巡检数据。
2. `_build_inspect_report()` 生成报告 ID、写入 JSON 并保存元数据。
3. 根据报告 ID 和 `APP_URL` 构造公开 PDF URL。
4. 调用 `send_feishu(..., report_url=public_pdf_url)`。
5. 飞书用户点击按钮后由公开路由动态生成完整 PDF。

### 分组定时或手动巡检

1. 按主机分组生成本组摘要和 AI 分析。
2. 使用本组巡检结果构建独立的 `type=inspect` 报告。
3. 在发送飞书消息前写入报告 JSON，并保存报告元数据。
4. 使用该分组报告 ID 构造公开 PDF URL。
5. 调用 `send_feishu_group_inspect(..., report_url=public_pdf_url)`。

所有分组推送入口复用同一个“构建并保存分组巡检报告”逻辑，避免定时任务、手动推送和批量生成路径产生不同链接行为。

### 报告页面手动推送

当被推送报告的 `type` 为 `inspect` 时，使用公开巡检 PDF URL；其他报告类型继续使用现有报告页面链接和权限规则。

## 保存与通知顺序

必须先成功保存报告 JSON，再构造链接并发送飞书通知。元数据保存继续沿用现有兼容策略，不因元数据数据库暂时不可用而阻断已经落盘的报告和飞书通知。

若报告 JSON 保存失败，则不发送带无效链接的通知，并把该分组或全局任务标记为失败。若只有 `APP_URL` 缺失，则继续发送不带按钮的摘要卡片。

## 访问控制与风险

公开路由只允许读取巡检报告，但不要求用户登录。任何获得或猜到链接的人都可能看到报告中的主机 IP、负责人、机房、磁盘和进程信息。

该风险是本需求已确认的取舍。接口仍需执行报告类型检查、报告 ID 校验和目录边界检查，不能把任意文件读取能力暴露为公开接口。

## 兼容性

- `report_url` 参数保持可选，现有未传链接的调用不会报错。
- 原有飞书摘要、关键词校验和 AI 内容截断保持不变。
- 原有登录态 PDF 导出和前端下载功能保持不变。
- 报告清理后，公开链接自然返回 404。

## 测试设计

### 卡片单元测试

- 完整巡检卡片传入 URL 时包含“查看完整巡检 PDF”按钮及正确 URL。
- 分组巡检卡片传入 URL 时包含同样的按钮及正确 URL。
- 两种卡片未传 URL 时都不包含 action 按钮。

### API 测试

- 有效巡检报告返回 `200`、`application/pdf` 和 `%PDF` 文件头。
- 不存在的报告返回 404。
- 日报和慢日志报告通过公开巡检路由访问时返回 404。
- 非法报告 ID 不能读取 `REPORTS_DIR` 外的文件。
- PDF 构建异常返回 500，并保留服务端错误日志。

### 推送流程测试

- 全局定时巡检在报告保存后把正确的公开 PDF URL 传给 `send_feishu()`。
- 分组推送在报告保存后把对应分组报告 URL 传给 `send_feishu_group_inspect()`。
- 报告页面手动推送巡检报告时使用公开 PDF URL。
- `APP_URL` 为空时继续发送不带按钮的巡检卡片。

### 回归验证

- 运行巡检报告、PDF、飞书通知、调度任务和报告路由相关后端测试。
- 执行变更文件的 Python 编译检查。
- 执行 `git diff --check`。

## 验收标准

- 飞书全局巡检消息包含可点击的“查看完整巡检 PDF”按钮。
- 飞书分组巡检消息包含可点击的“查看完整巡检 PDF”按钮。
- 点击后无需登录即可打开对应的完整巡检 PDF。
- 分组卡片不会链接到其他分组或全局报告。
- 非巡检报告不能通过公开巡检接口访问。
- 未配置 `APP_URL` 时不出现无效按钮，原巡检告警仍可发送。
- 报告超过保留期被清理后，旧链接返回 404。
