# CLI-WeChat-Bridge 二维码展示修复说明

## 修复背景

在 `192.168.9.221:8080` 的 Codex 微信接入控制台中，点击“生成二维码”后，页面无法正常展示二维码图片，导致用户无法直接扫码登录。

## 问题现象

- 页面点击“生成二维码”后，二维码区域为空白或加载失败
- 前端虽然拿到了接口返回值，但 `<img>` 无法正常渲染
- 用户只能看到扫码链接，不能直接在页面上扫二维码

## 根因分析

问题出在二维码接口返回的数据类型不对：

- `/api/qrcode` 返回的 `qrcodeImageUrl`
- 实际上是微信的扫码跳转链接
- 不是浏览器可直接作为二维码图片渲染的图片资源

前端当前逻辑是：

```html
<img id="qr-image" alt="微信二维码" />
```

并直接执行：

```js
el.qrImage.src = data.qrcodeImageUrl;
```

由于 `qrcodeImageUrl` 本质上只是一个页面链接，而不是图片地址，所以浏览器无法把它当作二维码图片显示。

## 修复方案

本次修复采用“服务端本地生成 SVG 二维码”的方式：

1. 保留原始扫码链接 `scanUrl`
2. 服务端将该链接编码成二维码
3. 生成 `data:image/svg+xml` 格式的数据 URL
4. 前端继续按原方式赋值给 `<img>`，无需改动页面调用逻辑

这样既兼容现有前端，又能保证二维码稳定展示。

## 本次修改文件

### 1. 新增二维码 SVG 生成模块

文件：

- `tools/CLI-WeChat-Bridge/web-console/lib/qr-svg.mjs`

作用：

- 基于现有依赖生成二维码矩阵
- 输出 SVG
- 转成 `data:image/svg+xml;charset=utf-8,...`

### 2. 修改二维码接口返回逻辑

文件：

- `tools/CLI-WeChat-Bridge/web-console/server.mjs`

修改内容：

- 引入 `buildQrSvgDataUrl`
- 在 `POST /api/qrcode` 中将原始扫码链接转换为 SVG 二维码
- 返回字段调整为：
  - `qrcodeImageUrl`: 页面可直接渲染的二维码图片
  - `scanUrl`: 原始微信扫码链接
  - `qrcodeSourceUrl`: 原始二维码来源链接

## 部署处理

由于线上服务已经在运行中，为了快速恢复可用性，本次采用热修方式处理：

1. 将修复后的源码同步到服务器
2. 把更新后的文件复制到运行中的容器 `codex-wechat-web`
3. 重启容器使修复生效

目标环境：

- 服务器：`192.168.9.221`
- 容器：`codex-wechat-web`
- 服务地址：`http://192.168.9.221:8080/`

## 修复结果

修复后，二维码接口返回内容已经变为浏览器可直接渲染的图片数据：

```json
{
  "qrcodeImageUrl": "data:image/svg+xml;charset=utf-8,...",
  "scanUrl": "https://liteapp.weixin.qq.com/...",
  "qrcodeSourceUrl": "https://liteapp.weixin.qq.com/..."
}
```

这意味着：

- 页面中的 `<img>` 可以直接显示二维码
- 用户可以在网页中直接扫码
- 原始扫码链接仍然保留，便于备用跳转或调试

## 验证结果

已完成以下验证：

- `GET /api/health` 正常返回
- `POST /api/qrcode` 正常返回二维码数据
- `qrcodeImageUrl` 已由原始链接变为 `data:image/svg+xml`
- 服务容器重启后正常运行

## 使用建议

如果页面仍然看不到最新二维码，建议：

1. 打开控制台页面  
   `http://192.168.9.221:8080/`
2. 执行浏览器强刷  
   `Ctrl + F5`
3. 重新点击“生成二维码”
4. 使用微信重新扫码

## 修复结论

本次问题不是微信接口失效，也不是前端样式问题，而是“二维码图片字段实际返回了链接而不是图片”。  
修复后，系统改为在服务端把扫码链接转换成 SVG 二维码，再返回给前端展示，页面扫码功能已恢复正常。
