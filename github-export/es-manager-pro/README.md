# ES Manager Pro

一个开箱即用的 **Elasticsearch 单文件管理工具**。

你只需要一个 `HTML` 文件，就可以在浏览器里完成多集群连接、节点巡检、索引管理、查询调试，以及日常 ES 运维操作。

这份 README 同时兼顾两种用途：

- 作为 GitHub 项目的说明文档
- 作为可直接改写成公众号文章的介绍文案

---

## 这是什么？

`ES Manager Pro` 是一个面向 Elasticsearch 日常管理场景的轻量前端工具。

它的特点很直接：

- **单文件**
- **零构建**
- **打开即用**
- **适合内网部署**

它不是一套复杂的后端系统，也不要求你先搭数据库、再配服务端、再做账号体系。

如果你的需求是：

- 临时连一个 ES 集群看看状态
- 快速查索引、查节点、查分片
- 需要一个比 Kibana Dev Tools 更偏运维视角的页面
- 想把多个 ES 环境放在一个界面里统一查看

那这个工具会非常顺手。

---

## 我为什么做这个工具？

很多团队在使用 Elasticsearch 时，都会碰到一个很现实的问题：

**查询有 Dev Tools，监控有监控平台，但缺一个“足够轻、足够快、足够顺手”的 ES 管理界面。**

尤其在下面这些场景里，这个问题会更明显：

- 开发、测试、预发、生产有多个 ES 集群
- 运维同学要快速看节点状态、索引规模、分片分布
- 开发同学要临时验证模板、别名、ILM、Pipeline
- 排查问题时，不想每次都手敲 API

所以就做了这个 **ES Manager Pro**。

## 这个工具能做什么？

### 1）多集群统一管理

支持保存和切换多个 Elasticsearch 集群，把开发、测试、预发、生产统一放到一个界面里管理。

### 2）集群概览与健康巡检

可以快速查看：

- 集群健康状态
- 节点数量
- 分片数量
- 文档规模
- 存储占用
- JVM / Heap 指标
- Pending Tasks 等运行信息

### 3）节点管理

查看节点列表、节点角色、节点资源状态，快速识别异常节点和热点节点。

### 4）索引管理

直接查看：

- 索引列表
- 索引状态
- 文档数量
- 主分片 / 副本分片
- 存储大小

适合做日常巡检和问题排查。

### 5）分片视图

分片问题是 ES 运维里最常见的一类问题。

工具内置分片视图，方便快速判断：

- 主分片 / 副本分片分布
- 未分配分片
- 初始化中的分片
- Relocating 分片

### 6）别名 / 模板 / ILM / Pipeline 管理

除了索引本身，还支持查看和管理：

- Alias
- Index Template
- ILM 生命周期策略
- Ingest Pipeline

### 7）数据浏览与查询控制台

内置轻量查询控制台，可直接发起 ES API 请求，适合：

- `_cluster/health`
- `_cat/indices`
- `_search`
- `_mapping`
- `_settings`

等常见调试动作。

### 8）快照、任务与集群设置

还包括：

- Snapshot 管理
- Task 管理
- Cluster Settings 查看与调整

## 快速开始

### 方式一：直接本地打开

核心文件就是：

```text
ES-Manager-Pro.html
```

双击即可打开。

### 方式二：用静态服务启动

更推荐通过一个简单静态服务访问：

```bash
python -m http.server 8080
```

然后浏览器访问：

```text
http://127.0.0.1:8080/ES-Manager-Pro.html
```

### 方式三：放到 Nginx / 静态服务器

直接把 `ES-Manager-Pro.html` 放到静态目录即可，例如：

```text
/usr/share/nginx/html/ES-Manager-Pro.html
```

## 第一次连接 Elasticsearch

打开页面后，新增集群并填写：

- 集群名称
- ES 地址
- 用户名 / 密码（如开启鉴权）
- 环境标识（开发 / 测试 / 生产）

连接成功后，连接信息会保存在当前浏览器本地存储中。

也就是说：

- **数据默认只保存在本地浏览器**
- **不会自动上传到第三方服务**

## 使用时要注意什么？

### 1）浏览器跨域（CORS）

因为这是浏览器直接请求 Elasticsearch API 的工具，如果你的 ES 没开跨域，请求可能失败。

常见解决方式：

- 给 ES 配置允许访问的 CORS
- 在前面加一层 Nginx / API Gateway 做反向代理

如果是生产环境，更建议通过内网网关或反向代理访问，而不是直接让浏览器裸连 ES。

### 2）不要暴露到公网

这是一个偏内部运维的工具，不建议直接暴露到公网。

更推荐放在：

- 公司内网
- VPN 后
- Zero Trust 访问体系后
- 堡垒机或受控跳板环境里

### 3）注意账号权限

页面上的操作会继承你连接 ES 所使用账号的权限。

建议：

- 只读场景使用只读账号
- 运维场景使用最小必要权限账号
- 不要默认使用超级管理员账号

## 适合怎么部署？

### 个人使用

本地打开，或者起一个简单静态服务就够了。

### 团队内部共享

推荐：

- Nginx 静态托管
- 配合 Basic Auth
- 放到公司内部门户或工具导航里

### 生产环境

推荐：

- 前置 Nginx / API Gateway
- 由反向代理转发到 ES
- 配访问控制
- 配 HTTPS
